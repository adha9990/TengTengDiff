# Git Commit 規範指南

你將協助我根據當前的變更生成符合規範的 Git 提交訊息。

## ⚠️ 重要規則 - 單次提交限制

**此命令僅處理當前時刻的變更，不會自動提交後續的修改：**

**單次執行原則**
   - 只處理執行 `/commit` 時已存在的變更
   - 不追蹤或提交後續的程式碼修改

## 提交訊息格式

```
<emoji> <type>(<scope>): <subject>

<body>

<footer>
```

### 格式說明
- **Header** (必填): 簡潔描述變更內容，長度控制在 72 字元內
- **Body** (選填): 詳細說明變更背景與內容，每行約 72 字元
- **Footer** (選填): 關聯 issue、breaking changes 等資訊

## Type 類型與對應 Emoji

| Emoji | Type     | 說明                                    |
|-------|----------|----------------------------------------|
| ✨     | feat     | 新功能                                  |
| 🐞     | fix      | Bug 修正                                |
| 📝     | docs     | 文檔更新                                |
| 🎨     | style    | 程式碼格式調整（不影響功能）              |
| 🦄     | refactor | 重構（既非新功能也非修復 bug）            |
| 🧪     | test     | 新增或修改測試                          |
| 🐳     | chore    | 構建過程或輔助工具的變更                  |
| ⚡     | perf     | 性能優化                                |
| 🔧     | config   | 配置文件修改                            |
| 🚀     | deploy   | 部署相關                                |

## 範例

### 簡單提交
```
✨ feat(auth): 新增使用者登入功能
```

### 包含詳細說明
```
🦄 refactor(model): 重構模型載入機制

- 將 autoloader 重命名為 providerMap 提升語意清晰度
- 統一使用 providerMap 作為變數名稱
- 更新所有相關引用路徑

這次重構改善了程式碼的可讀性和維護性
```

### 修復 Bug
```
🐞 fix(upload): 修復大檔案上傳失敗問題

問題原因：
- 上傳超時設定過短
- 缺少錯誤重試機制

解決方案：
- 調整超時時間為 5 分鐘
- 新增自動重試機制（最多 3 次）

Closes #123
```

## 撰寫原則

1. **Header 部分**
   - 使用動詞開頭（新增、修復、更新、重構等）
   - 簡潔明瞭，避免冗詞贅字
   - scope 使用小寫英文，代表影響範圍

2. **Body 部分**
   - 說明「為什麼」而非「怎麼做」
   - 可使用項目符號整理要點
   - 段落間保留空行

3. **Footer 部分**
   - 關聯 issue：`Closes #123`、`Fixes #456`
   - Breaking changes：`BREAKING CHANGE: 描述`

## 注意事項

- 提交訊息使用**中文**撰寫
- 每行長度控制在 **100 字元**以內
- 保持訊息簡潔但資訊完整
- 遵循團隊既定的 scope 命名慣例

## 🛑 執行限制

**重要：此命令為單次記憶模式**

1. **不會記憶先前的提交內容**
   - 每次執行都是獨立的操作
   - 不會延續之前的提交狀態

2. **不會自動處理後續變更**
   - 只處理執行命令當下的變更
   - 後續修改需要重新執行 `/commit`
